Docker

1) What Problem Does Docker Solve? 

   Imagine that we create a web application on our computer. To run correctly, our application may require:

	* a specific Linux distribution
	* a specific version of PHP
	* MariaDB
	* NGINX
	* configuration files
	* environment variables
	* libraries and dependencies

   If another person tries to run the same application on a different computer, it may fail because their 
   environment is different.

   Docker solves this problem. Docker packages an application together with everything it needs to run.
   The application will then behave the same way on every machine that has Docker installed.

2) What Is Docker?

   Docker is a platform used to create, package, run, and manage applications inside isolated environments called containers.
   A container is a controlled environment where an application can run with everything it needs: its files, libraries, 
   dependencies, configuration, and runtime.
   Instead of installing an application directly on the host operating system, Docker allows us to place that application 
   inside a container and run it there.

   The main idea is simple: Application + Dependencies + Configuration = Containerized Application

   This means the application does not depend heavily on what is installed on the host machine. The host only needs Docker.
   For example, imagine a WordPress project. WordPress needs PHP, PHP-FPM, configuration files, permissions, and a connection 
   to a database. Without Docker, we would install and configure everything directly on the system. With Docker, WordPress can 
   run inside its own container, MariaDB can run inside another container, and NGINX can run inside another container.

												Host Machine
												│
												├── NGINX Container
												├── WordPress Container
												└── MariaDB Container

   Each container has its own responsibility. Because containers are isolated, they do not interfere with the rest of the system.

   Docker is not only one command. It is a complete system composed of several parts.
   
   The most important parts are: Docker Client
								 Docker Daemon
								 Docker Images
								 Docker Containers
								 Dockerfiles
								 Docker Volumes
								 Docker Networks
								 Docker Compose

    Together, these components allow us to build and run complete infrastructures. When we type: docker run nginx , we are not 
	simply opening a program. We are asking Docker to create a container from an image and start a process inside that isolated environment.

3) Docker Client

   The Docker Client is the command-line tool that we use in the terminal. It is the command: docker
   When we write: docker ps
				  docker build
				  docker run
				  docker images

   We are using the Docker Client. The client does not do the heavy work itself. It sends requests to the Docker Daemon.

   Example: User -> docker command -> Docker Client -> Docker Daemon

   The client is the interface. The daemon is the engine.

4) Docker Daemon

   The Docker Daemon is the background service responsible for doing the real work. It manages: Images
																								Containers
																								Networks
																								Volumes
																								Builds
																								Execution

   The daemon is usually called: dockerd
   When the Docker service is running on Linux, the daemon is active in the background.
   Example: sudo systemctl status docker
   If the daemon is stopped, Docker commands will fail because the client has nobody to talk to.
   So when we execute: docker run my_image, the client sends the request to the daemon, and the daemon creates and starts the container.


4) What Is a Docker Image?

   A Docker Image is a blueprint. Think of it as a recipe. Example: Recipe:
   																	  - Install Debian
																	  - Install MariaDB
																	  - Copy configuration
																	  - Copy startup script

   This recipe is written inside a Dockerfile. Docker Image is a template for creating Docker containers.
   It provides all the necessary instructions and dependecies required to create and run a containerized 
   application.
   Docker reads the Dockerfile and creates an image. The image itself does not run. It is only a template.

5) What Is a Docker Container?

   Docker Container is a lightweight, standalone, and executable software package that encapsulates an 
   application and its dependencies, ensuring consistent and reliable execution across different computing 
   environments. It provides a standardized approach to package and deploy applications, allowing them to 
   run in isolation with their own filesystem, libraries, and configuration settings.

   A container is a running instance of an image. The image is the blueprint. The container is the actual 
   running application.

   Similar example:  Class      → Image
                     Object     → Container

   One image can create multiple containers.

6) Dockerfile

   A Dockerfile contains instructions for building an image.

   Example: FROM debian:bookworm

			RUN apt-get update
			RUN apt-get install -y mariadb-server

			COPY init.sh /bin/init.sh

			ENTRYPOINT ["/bin/init.sh"]

   Docker executes these instructions step by step. The result is a Docker image.

7) Building an Image

   When Docker reads a Dockerfile: docker build .

   Docker performs:      Read Dockerfile
        				        ↓
						 Execute Instructions
                                ↓
                           Create Image

   The image is then stored locally. We can see all images: docker images.

8) Running a Container

   Once the image exists:  docker run myimage

   Docker creates a container from that image. The container then starts executing its main process.

9) Docker Daemon (Dockerd)

   The Docker daemon is the engine behind Docker. Is the core component of DOcker platform. It is responsible
   for building, running, and managing Docker containers. It is always running in the background.
   Whenever we execute: docker build / docker run / docker ps, we are actually talking to the Docker daemon.
   The daemon performs all the real work.

10) Docker Client

    The Docker client is the command line tool:  docker

11) Docker Compose

	Running one container is easy. Running several containers becomes difficult.
    Each needs: * ports
				* volumes
				* networks
				* environment variables

	Docker Compose allows all services to be described inside a single file: docker-compose.yml

    Example: docker compose up

    Docker automatically creates and starts all containers. This is exactly what Inception requires.
