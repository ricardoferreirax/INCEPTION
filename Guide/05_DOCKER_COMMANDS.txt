DOCKER COMMANDS

	1) Build Commands 
	
	   1.1) Build All Images: docker compose build

	        This command reads the docker-compose.yml file and builds all services that have a build: section.
			For example, if the Compose file has:

						services:
  						  mariadb:
  						  wordpress:
  						  nginx:

			then this command tries to build all three images. We use it when we want to prepare all project 
			images without starting the containers.

	   1.2) Build only one service:  docker compose build mariadb 
	        
			This builds only the mariadb image. We use this when we are working only on MariaDB and we do not 
			want to waste time rebuilding WordPress or Nginx.

			Other examples:  docker compose build wordpress
							 docker compose build nginx

	   1.2) Build Without Cache:  docker compose build --no-cache mariadb

	        Docker normally uses cache. That means if a Dockerfile step has not changed, Docker reuses the 
			previous result. The --no-cache flag disables that behavior and forces Docker to rebuild every 
			step from zero.
			We use this when: We changed the Dockerfile but Docker seems to ignore it.
							  We changed an install command.
							  We changed scripts copied into the image.
							  The image behaves like an old version.


	2) Starting Containers 

	   2.1) Start All Services:  docker compose up 

	        TThis starts all services and attaches the terminal to the logs. Good for debugging because we 
			immediately see errors. If we press CTRL + C the containers stop.

	   2.2) Start One Service only:  docker compose up mariadb

	        This starts only the mariadb service. Useful while testing MariaDB alone.

	   2.3) Start All The Services And Rebuild First:  docker compose up --build

	        This builds the images first and then starts the containers.
            We use this after changing: Dockerfile
										scripts
										config files
										docker-compose.yml

	   2.4) Start One Service And Rebuild It: docker compose up --build mariadb

			This rebuilds and starts only MariaDB. Very useful during development.

	   2.5) Start containers in background:  docker compose up -d

			The -d means detached mode. The containers run in the background and the terminal becomes free.
			We use this when the project is already working.

			For only MariaDB:  docker compose up -d mariadb

			Difference: docker compose up        shows logs directly and stops with CTRL + C and
                        docker compose up -d     runs in background.


	  3) Stopping Containers 

	  	 3.1) Stop Containers But Keep Them (Without Deleting Anything):  docker compose stop

			  This stops the containers but does not remove them.
			  We can start them again with:  docker compose start
			  This is useful when we want a temporary pause.

		 3.2) Stop And Remove Containers:  docker compose down

			  This stops and removes the containers and the default compose network.
              It does not delete named volumes by default.
			  THis is the normal command to cleanly stop the project.
              We use this normally when restarting the project.

		 3.3) Stop And Remove Containers + Volumes:  docker compose down -v

			  The -v flag means volumes and remove containers, networks, and named volumes.
			  We use this when we want to reset persistent data.
              For Inception, this is useful when we want to reset MariaDB completely.

			  However, because the volume is a bind mount /home/rmedeiro/data/mariadb we may also 
			  need: sudo rm -rf /home/rmedeiro/data/mariadb/*  .

			  Important difference: docker compose down                         keeps database files.
                                    docker compose down -v                      removes named Docker volumes.
                                    sudo rm -rf /home/rmedeiro/data/mariadb/*   removes the real MariaDB files from the 
									                                            VM host directory.

			  Difference: docker compose down keeps the database data, otherwise, docker compose down -v
                          removes the Docker volume. But if our volume is a bind mount to /home/rmedeiro/data/mariadb
                          we may also need to delete the files manually.


	  4) Listing Containers 

	  	4.1) Show Running Containers:  docker ps 

		     THis only shows containers that are currently running.

			 If they are running we see something like:  mariadb   Up
														 wordpress Up
														 nginx     Up

		4.2) Show All Containers:  docker ps -a 

			 This shows both running and stopped containers. Useful when something stopped unexpectedly.

			 Example: Exited (0) means the process stopped successfully.

			 Example: Exited (1) usually means an error.

			 Example: Restarting means the container is crashing and Docker keeps restarting it.

	  5) Logs 

	    5.1) Show Logs For All Services:  docker compose logs

		     THis prints logs from every service in compose.

		5.2) Follow Logs Live: docker compose logs -f mariadb

		     The -f means follow. Docker keeps the log open and prints new lines as they appear.
             Useful when testing startup.

		5.3) Docker Direct Logs:  docker logs mariadb 

		     THis uses the container name directly.

			 Difference: docker compose logs mariadb    uses the Compose service name.
                         docker logs mariadb            uses the real container name.

			Both work because I set: container_name: mariadb


	  6) Containers Cleanup

	     6.1) Remove Stopped Containers:  docker container prune 

		      THis deletes stopped containers. It does not delete running containers.

		 6.2) Remove One Container: docker rm mariadb 

		      If it is running:  docker stop mariadb 
			                     docker rm mariadb 

		      Or force:  docker rm -f mariadb


	  7) Images 

	     7.1) List Images:  docker images    or   docker image ls

		      Show local Docker images.

		 7.2) Remove One Image:  docker rmi mariadb
		     
			  If it is being used by a container:  docker compose down
												   docker rmi mariadb

		 7.3) Remove Unused Images:  docker image prune

		      THis removes dangling/unused images.

			  More aggressive:  docker image prune -a 

			  The -a removes all unused images.


	  8) Volumes 

	     8.1) List Docker Volumes:  docker volume ls 

		 8.2) Inspect A Volume:  docker volume inspect mariadb 

		      This shows details about the volume and how the volume is configured. 
			  From bind mount, we should see something like:

			  "Options": {
  				"device": "/home/rmedeiro/data/mariadb",
  				"o": "bind",
  				"type": "none"
			   }

			   This means Docker stores MariaDB data in /home/rmedeiro/data/mariadb, inside the Debian VM.

		 8.3) Remove A Volume:  docker volume rm mariadb 

		      If the volume is in use:  docker compose down
										docker volume rm mariadb

		 8.4) Remove Unused Volumes:  docker volume prune 

		      THis removes unused volumes.

	  9) Networks

	     9.1) List Networks:  docker network ls 

		      Compose creates a network. It may be called srcs_inception

		 9.2) Inspect Network:  docker network inspect srcs_inception
		     
			  This shows which containers are attached to the network.
			  Useful to verify that mariadb, wordpressnginx

			  are all on the same network.




	  6) Deleting The Whole Data Directory (Bind Mount Host Directories)

	     For Inception, the subject expects data under: /home/login/data

		 Create the directories:  mkdir -p /home/rmedeiro/data/mariadb
								  mkdir -p /home/rmedeiro/data/wordpress

		 Check ownership:  ls -ld /home/rmedeiro/data
						   ls -ld /home/rmedeiro/data/mariadb
						   ls -ld /home/rmedeiro/data/wordpress

		 If permissions are wrong:  sudo chown -R rmedeiro:rmedeiro /home/rmedeiro/data

		 If we need to delete all data:  docker compose down -v
										 sudo rm -rf /home/rmedeiro/data
										 mkdir -p /home/rmedeiro/data/mariadb
									     mkdir -p /home/rmedeiro/data/wordpress


	  6) Entering A Container and SQL Commands

	     6.1) Enter MariaDB Container:  docker exec -it mariadb sh

			  Meaning:   docker exec   run a command inside an existing container
							  -i       interactive mode
						 	  -t       allocate a terminal
							mariadb    container name
							  sh       shell to open

			  This only works if the container is running.

		 6.2) Connect as root:  mariadb -u root -p 

		       It will ask for the password (the password inside secrets/db_root_password.txt).

		 6.3) Show Databases:  SHOW DATABASES;

		      Expected:  information_schema
						 mysql
						 performance_schema
						 sys
						 wordpress

			  The important one is: wordpress

		 6.4) Show Users: SELECT User, Host FROM mysql.user;

		      Expected example:   root      localhost
								  rmedeiro   %

		 6.5) Check Privileges: SHOW GRANTS FOR 'rmedeiro'@'%';
