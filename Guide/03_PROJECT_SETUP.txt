Configure the Local Domain (username.42.fr)

The Inception subject requires the website to be accessible through: user.42.fr  (e.g: rmedeiro.42.fr)

However, this domain does not exist on the public internet. To make the operating system recognize it, we must 
manually create a local DNS entry using the /etc/hosts file.

What is /etc/hosts ? The /etc/hosts file is a local DNS table.
Before contacting an external DNS server, Linux checks this file to see if the domain name is already known.

Example:   rmedeiro.42.fr
                 |
				 v
             127.0.0.1

1) Edit the Hosts File - open the hosts file:  sudo vim /etc/hosts

   We will see something similar to:  127.0.0.1 localhost
                                      127.0.1.1 debian

2) Add 42 Domain - add the following line:  127.0.0.1 rmedeiro.42.fr
   
   The file should look similar to:  127.0.0.1 localhost
                                     127.0.0.1 rmedeiro.42.fr
                                     127.0.1.1 debian

3) Check the Configuration:  ping rmedeiro.42.fr

   Expected output: PING rmedeiro.42.fr (127.0.0.1)


How Does It Work? When the browser requests https://rmedeiro.42.fr , Linux performs:  rmedeiro.42.fr
																						   |
																						   v
																					   127.0.0.1
																					       |
																						   v
																					 NGINX Container
																					       |
																						   v
																						WordPress
																						   |
																						   v
																						MariaDB

No public DNS server is involved. Everything happens locally inside the VM.

The /etc/hosts file acts as a local DNS table.
When the browser requests https://rmedeiro.42.fr the operating system automatically translates: rmedeiro.42.fr → 127.0.0.1
which points to the local machine.
This allows NGINX to receive requests for the domain required by the subject without needing a real public DNS record.

If you access the website from your host machine (outside the VM), you must also add 127.0.0.1 rmedeiro.42.fr to the host 
machine's hosts file.
Otherwise, the browser running on the host will not be able to resolve the domain name.

Create Volumes on the Host Machine

One of the requirements of the Inception subject is to use Docker named volumes for persistent storage.
The subject explicitly states that the volume data must be stored inside /home/login/data , where login is our 42 login.

Where Should These Directories Be Created? Although we are running Debian inside VirtualBox, the Debian VM is considered 
the Docker host machine because Docker Engine is installed and running inside the VM. Therefore, when the subject says 
/home/login/data on the host machine it means inside the Debian virtual machine. Docker never sees our physical computer. 
Docker only sees the Debian VM where it is installed.

Create the folders:  mkdir -p /home/rmedeiro/data/mariadb
                     mkdir -p /home/rmedeiro/data/wordpress

Check that they were created successfully:  ls -la /home/rmedeiro/data

Why Do We Need These Directories? Containers are designed to be temporary. If a container is remover, for example,
docker rm mariadb, it completely removes the container and its internal filesystem.
Without persistent storage, MariaDB database, WordPress files, uploaded content, users and posts would be lost.
So, MariaDB and WordPress need persistent data. With volumes the data is preserved.
The containers can be destroyed and recreated while the date remains safely stored inside /home/rmedeiro/data/mariadb
and /home/rmedeiro/data/wordpress .
This way, if containers are deleted and recreated, the data is not lost.
This is exactly why the Inception subject requires persistent volumes.

Create the .env file

The .env file stores environment variables used by Docker Compose and by the other containers, such as MariaDB, WordPres 
and NGINX.

Instead of hardcoding configuration values directly inside docker-compose.yml, Dockerfiles, shell scripts or nginx.conf, we 
define them once in .env file and reuse them throughout the project.
This makes the project easier to maintain, to configure and safer from accidental hardcoded values.

The .env must be located inside the srcs folder:  vim srcs/.env

What must be inside .env? For the mandatory part, we need, for example: DOMAIN_NAME
										                                MYSQL_DATABASE
										  				                MYSQL_USER
										                                WP_TITLE
										                                WP_ADMIN_USER
										                                WP_ADMIN_EMAIL
										                                WP_USER
										                                WP_USER_EMAIL

How Does .env Work? How does, for example, WordPress know the value of MYSQL_USER?

Inside .env, we define the variable MYSQL_USER=wp_user.
Inside docker-compose.yml we have: 		services:
  										  wordpress:
                                            env_file:
                                              - .env

Docker Compose automatically reads the .env file and loads its variables.															
When the container starts, Docker Compose exports these variables into the container environment.
Inside the container echo $MYSQL_USER returns wp_user.

Why Is This Useful? Suppose we change MYSQL_USER=wp_user to MYSQL_USER=my_new_user.
No Dockerfile, script or configuration file needs to be modified.
After rebuilding the containers (docker compose up --build) the new value is automatically propagated 
everywhere.

What should NOT be inside .env? 

Passwords should not be hardcoded inside source files. We avoid: MYSQL_PASSWORD=1234
																 WP_ADMIN_PASSWORD=1234

Inside Dockerfiles, shell scripts, wp-config.php and Git repository.
Instead, store sensitive values using Docker secrets: secrets/db_root_password.txt
													  secrets/db_password.txt
													  secrets/wp_admin_password.txt
													  secrets/wp_user_password.txt

and read them at runtime.

What Should NOT Be Hardcoded? Avoid writing configuration values directly inside Dockerfiles, setup scripts,
wp-config.php, nginx.conf and docker-compose.yml. For example, we avoid: MYSQL_DATABASE=wordpress
                                                                         MYSQL_USER=wp_user

inside scripts.
Instead, we read the values from environment variables.
Passwords should be stored in Docker secrets instead. Why? The subject says credentials, passwords and API keys 
should not be stored directly in the Git repository. The .env file is useful for configuration, but it is not 
ideal for secrets.

Why Use Environment Variables? Environment variables allow the same codebase to be reused with different configurations.
For example: DOMAIN_NAME=rmedeiro.42.fr can be used simultaneously by Docker Compose, NGINX, WordPress and SSL certificate 
generation without modifying any source file.
Changing the domain only requires editing one line.

Important Evaluation Note: The evaluation sheet explicitly states that credentials, API keys and environment variables must be 
configured during the evaluation. For this reason: srcs/.env and secrets/ should not be committed to the Git repository.
We have to add them to: srcs/.env and secrets/ and create them locally during evaluation.

Create Docker Secrets

Docker Secrets are used to store sensitive information such as passwords outside of the source code.
Instead of hardcoding passwords directly inside we store them in dedicated files and make them available to containers 
at runtime.
This improves security and follows the recommendation of the Inception subject, which explicitly states that credentials and 
sensitive information should not be exposed in the repository.

If it does not already exists, we create the Secrets Directory:  mkdir -p secrets

We create the following secret files: echo "<DB_ROOT_PASSWORD>" > secrets/db_root_password.txt
						              echo "<DB_PASSWORD>" > secrets/db_password.txt
						              echo "<WP_ADMIN_PASSWORD>" > secrets/wp_admin_password.txt
						              echo "<WP_USER_PASSWORD>" > secrets/wp_user_password.txt

What each secret is for?

-> db_root_password.txt: Contains the password for the MariaDB root user account.
-> db_password.txt: Contains the password for the normal MariaDB user used by WordPress.
                    WordPress uses this account to connect to MariaDB, create tables, read data and write data.
-> wp_admin_password.txt: Contais the password for the WordPress administrator account.
-> wp_user_password.txt: Contains Password for the normal WordPress user.

How Do Docker Secrets Work? How does WordPress access the database password if it is not inside the code?

Example: 

secrets/db_password.txt is declared inside:  secrets:
                                               db_password:
                                                 file: ../secrets/db_password.txt

and attached to a service:  services:
                              wordpress:
                                secrets:
                                  - db_password

Inside the container Docker automatically creates /run/secrets/db_password .
The initialization script can then read it DB_PASSWORD=$(cat /run/secrets/db_password) without exposing 
the password anywhere in the source code.

Set Secure Permissions on Secrets:  chmod 600 secrets/*

What does chmod 600 mean? 6 - owner can read and write
						  0 - group has no permissions
                          0 - others have no permissions

So only the owner can read and edit the files.
In practice: -rw------- db_password.txt

Why Do We Use chmod 600? Secrets contain sensitive information. Restricting permissions prevents accidental 
exposure, unauthorized access and other users on the system reading passwords.
Even though Inception is developed inside a local VM, using restrictive permissions is considered good practice 
and reflects real-world server security practices.

Create .gitignore

The .gitignore file tells Git which files and directories should not be tracked by the repository.
This is especially important for Inception because some files contain passwords, credentials, machine-specific 
configuration and local development data. These files should never be pushed to GitHub.

At the root of the project, we create the .gitignore: vim .gitignore and add: secrets/ and srcs/.env  .

Why Ignore secrets/? The secrets/ directory contains passwords. These files contain sensitive information and must 
never be committed to the repository. If they are pushed to GitHub, anyone with access to the repository can read 
the passwords. Ignoring this directory ensures that credentials remain local to the machine where the project is being 
executed.

Why Ignore srcs/.env? The .env file contains environment variables. Even if passwords are stored in Docker Secrets, the 
.env file may still contain machine-specific configuration such as DOMAIN_NAME, database names, usernames and email addresses.
Keeping the file local makes the project easier to configure on different machines.

Important Evaluation Note: The evaluation sheet explicitly states that credentials, API keys and environment variables
must be configured during the evaluation. For this reason, srcs/.env and secrets/ should not be present in the Git repository.
Instead, they should be created locally before running the project.

Recommended Practice: A solution is to provide an example file srcs/.env.example , which contains the expected variables without 
any sensitive values.

Example:

DOMAIN_NAME=<DOMAIN_NAME>
MYSQL_DATABASE=<MYSQL_DATABASE>
MYSQL_USER=<MYSQL_USER>
WP_ADMIN_USER=<WP_ADMIN_USER>
WP_ADMIN_EMAIL=<WP_ADMIN_EMAIL>
WP_USER=<WP_USER>
WP_USER_EMAIL=<WP_USER_EMAIL>

This allows other users or evaluators to understand which variables are required while still keeping the real .env private.
