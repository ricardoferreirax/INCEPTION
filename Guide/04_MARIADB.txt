MariaDB Service

The MariaDB container is the first service implemented in the Inception project. Its purpose is to provide a 
database server that will later be used by the WordPress container.
The MariaDB service is composed of three files:

								requirements/mariadb/
								├── Dockerfile
								├── conf/
								│   └── 50-server.cnf
								└── tools/
									└── init_mariadb.sh

MariaDB is the database service used by WordPress, and it does not depend on any other service, so it 
is the first container that can be started.
The image is built from the own Dockerfile located in: ./requirements/mariadb/Dockerfile
The database files are stored in a Docker named volume mounted at: /var/lib/mysql. That path is connected
to a Docker named volume. The real persistent data is stored on the VM host at: /home/rmedeiro/data/mariadb

Volumes are used to persist data outside the container filesystem. As docker containers are temporary, if a 
container is deleted, everything stored inside it (its internal filesystem) is deleted as well.

MariaDB stores all database files inside /var/lib/mysql. To avoid losing the database when containers are 
rebuilt or recreated, this directory is connected to a Docker named volume.

Docker containers are isolated from each other. To allow the services to communicate, they must be connected
to the same Docker network. All containers will belong to this network. This means that a container can reach 
another container using its service name. 
The subject requires a Docker network that connects all services.

Example: WordPress connects to mariadb:3306, where: mariadb -> service name
                                                    3306    -> MariaDB port

Docker automatically resolves "mariadb" to the IP address of the MariaDB container.

Dockerfile

	The Dockerfile is responsible for building the MariaDB image. It starts from a Debian base image, installs 
	MariaDB Server, copies the configuration files and defines the startup script executed when the container starts.

	A Dockerfile is a text file that contains instructions for building a Docker image. We can think of it as a recipe.

	For example:  FROM debian:bookworm
				  RUN apt-get update && apt-get install -y mariadb-server
				  COPY tools/init_mariadb.sh /bin/init_mariadb.sh
				  ENTRYPOINT ["/bin/init_mariadb.sh"]

	means:  Start from Debian.
			Install MariaDB.
			Copy my startup script into the image.
			When the container starts, execute that script.

	The Dockerfile does not run the final service directly while building. It creates an image. Later, Docker uses that 
	image to create a container.

	A Dockerfile creates an image.

	Dockerfile  →  docker image  →  docker container

	Example: docker compose build mariadb reads the Dockerfile and creates the image mariadb, then docker compose up mariadb
             creates and starts a container from that image.

	So:  Image     =  blueprint/template
		 Container =  running instance

	For Inception, requirements/mariadb/Dockerfile is the blueprint for MariaDB container.

	Why is the Dockerfile important?

    In Inception, the subject requires us to build our own service images. We cannot simply use ready-made DockerHub images like:

	image: mariadb
	image: wordpress
	image: nginx

	Instead, we need something like:

	build:
	  context: ./requirements/mariadb
	  dockerfile: Dockerfile

	That means: Build my own MariaDB image from my own Dockerfile.

	The Dockerfile is important because it defines: Which operating system the service uses
													Which packages are installed
													Which configuration files are added
													Which scripts are copied
													Which permissions are applied
													Which command runs when the container starts

	For MariaDB, the Dockerfile prepares the environment, but the init_mariadb.sh script does the runtime initialization.


	1) Base Image
	
	   FROM debian:bookworm

	   Every Dockerfile needs a starting point. This line says: use Debian Bookworm as the base operating system.
	   So our image starts with a minimal Debian filesystem. It give us things like: /bin
																					 /etc
																					 /usr
																					 apt-get
																					 basic Linux directories

       But it does not automatically include MariaDB. We install later.
       In Inception, this is important because the subject asks us to use Debian or Alpine, usually the penultimate 
	   stable version depending on the subject version.

	2) Install MariaDB

	   RUN apt-get update && apt-get install -y mariadb-server

	   The command RUN executes a command while building the image.
	   
	   What does apt-get update do? It does not update installed programs. It updates the local list of available packages.
       Without it, Debian may not know where to download mariadb-server from.

       What does apt-get install -y mariadb-server do? It installs the MariaDB server package.

	3) Copy MariaDB config

	   COPY conf/50-server.cnf /etc/mysql/mariadb.conf.d/50-server.cnf

	   The command COPY copies a file from project into the image.
       
	   Source:                    conf/50-server.cnf
	   Destination inside image:  /etc/mysql/mariadb.conf.d/50-server.cnf

	   This file configures MariaDB.

	4) Copy initialization Script

	   COPY tools/init_mariadb.sh /bin/init_mariadb.sh

	   This copies the custom startup script into the image.

	   Source in project: 			  tools/init_mariadb.sh
	   Destination inside container:  /bin/init_mariadb.sh

	   This script is responsible for runtime setup: creating /run/mysqld
													 fixing permissions
													 reading Docker secrets
													 initializing /var/lib/mysql
													 creating the WordPress database
													 creating the WordPress database user
													 starting mysqld

	   This is not done directly in the Dockerfile because some information is only available when the container starts, 
	   for example: Docker secrets, environment variables and mounted volume contents.

	   A Dockerfile runs at image build time. The script runs at container runtime.

    5) Set Permissions And Create Directories

	   RUN chmod +x /bin/init_mariadb.sh && mkdir -p /run/mysqld /var/lib/mysql \
		   && chown -R mysql:mysql /run/mysqld /var/lib/mysql


	   Make the script executable: chmod +x /bin/init_mariadb.sh, that gives us the execution permission to init_mariadb.sh script.
	   Without this, Docker may fail with permission denied, because the file exists but cannot be executed.
	   The +x add executable permission, so the script can run as a program.

	   Create needed directories: mkdir -p /run/mysqld /var/lib/mysql, that creates /run/mysqld  and   /var/lib/mysql .
	   THe -p means do not fail if the directory already exists. Create parent directories if needed.
	   MariaDB needs /run/mysqld for runtime files like the socket and PID file, and needs /var/lib/mysql for database files.

	   Give ownership to mysql user: chown -R mysql:mysql /run/mysqld /var/lib/mysql, that changes owner and group of the directories.
	   mysql:mysql means: owner = mysql
					      group = mysql
	   The -R means recursive.
	   MariaDB should not run as root. It should run as the mysql user. So this makes sure the mysql user can write to: /run/mysqld
       and /var/lib/mysql . This matters because MariaDB needs to create files like: /run/mysqld/mysqld.sock
																					 /var/lib/mysql/ibdata1
																					 /var/lib/mysql/mysql/

	   Without correct permissions, MariaDB may fail with: permission denied.

	6) Expose Port 

	   EXPOSE 3306

	   This documents that the container listens on port 3306. MariaDB uses port 3306.

    7) Entrypoint 

	   ENTRYPOINT ["/bin/init_mariadb.sh"]

	   This defines the command that runs when the container starts.
       When we do: docker compose up mariadb , Docker creates a container from the image and runs: /bin/init_mariadb.sh .
	   The script then eventually runs: exec mysqld --user=mysql --datadir="$DATADIR".
	   So the real final process becomes: mysqld .
	   This is important because a Docker container stays alive only while its main process is running.
       If the entrypoint script ends without starting a running process, the container stops.
       That is why the script must end with: exec mysqld ...


Build time vs runtime

This is one of the most important concepts. Dockerfile runs at build time.

These lines run when we build the image:  RUN apt-get update && apt-get install -y mariadb-server
										  COPY conf/50-server.cnf /etc/mysql/mariadb.conf.d/50-server.cnf
										  COPY tools/init_mariadb.sh /bin/init_mariadb.sh
										  RUN chmod +x ...

Command: docker compose build mariadb

At this moment, Docker does not yet run the real database. It only prepares the image.

The Entrypoint runs at runtime. This line defines what runs later: ENTRYPOINT ["/bin/init_mariadb.sh"].

It runs when we start the container: docker compose up mariadb

At runtime, the container has access to:  environment variables from .env
										  Docker secrets in /run/secrets
										  mounted volume /var/lib/mysql
										  Docker network

That is why passwords and database initialization belong in the script, not directly in the Dockerfile.
