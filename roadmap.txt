Docker Engine installation

	> Update System: sudo apt-get update && sudo apt-get upgrade -y

	> Install Required Depedencies: sudo apt-get install ca-certificates curl gnupg -y

      Installs the packages required to securely download and verify Docker packages: -> ca-certificates: HTTPS certificate validation.
																					  -> curl: Download files from the internet.
																					  -> gnupg: Verify package signatures using GPG keys.

	> Create Keyrings Directory: sudo install -m 0755 -d /etc/apt/keyrings

      Creates the directory where trusted repository keys will be stored.

	> Add official Docker key: curl -fsSL https://download.docker.com/linux/debian/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

      Downloads Docker's official signing key and converts it to the binary GPG format required by APT.

	> Set Key Permissions: sudo chmod a+r /etc/apt/keyrings/docker.gpg

	  Allows the APT package manager to read the Docker GPG key.

	> Add Docker repository: echo "deb [arch="$(dpkg --print-architecture)" signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/debian \
							"$(. /etc/os-release && echo "$VERSION_CODENAME")" stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

	  Adds Docker's official repository to the APT sources list.

	> Update packages: sudo apt-get update

	> Install Docker Engine: sudo apt-get install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

      Installs: -> docker-ce: Docker Engine.
				-> docker-ce-cli: Docker command-line interface.
				-> containerd.io: Container runtime.
				-> docker-buildx-plugin: Advanced image building support.
				-> docker-compose-plugin: Docker Compose v2 (docker compose).

	> Start Docker Service: sudo systemctl start docker

	  Starts the Docker daemon immediately.

	> Enable Docker at Boot: sudo systemctl enable docker

	  Ensures Docker starts automatically every time the system boots.

	> Add User to Docker Group: sudo gpasswd -a $USER docker

	  Allows the current user to run Docker commands without using sudo.

	> Apply Group Changes: newgrp docker

	  Applies the new group membership without requiring a logout or reboot.

	> Verify Docker Installation: docker run hello-world

	  Downloads and runs Docker's test container. Expected output: Hello from Docker!
      If this message appears, Docker is installed correctly.

	> Verify Docker Compose: docker compose version

      Checks that Docker Compose v2 is installed and available.

	> Check Docker Service Status: systemctl status docker

	  Displays the current state of the Docker daemon.
      Expected status: active (running)

    > Display Docker Information: docker info

	  Displays detailed information about: - Docker version
										   - Storage driver
										   - Available networks
										   - Available volumes
										   - Docker root directory
										   - Running containers

	  This command is useful to confirm that the Docker daemon is fully operational.

Configure Local Domain (username.42.fr)

The Inception subject requires the website to be accessible through: user.42.fr  (e.g: rmedeiro.42.fr)

Since this domain does not exist on the internet, we must manually map it to our local machine using the  
/etc/hosts file.

1) Edit the Hosts File - open the hosts file:  sudo vim /etc/hosts

   You will see something similar to: 127.0.0.1 localhost
                                      127.0.1.1 debian

2) Add Your 42 Domain - add the following line: 127.0.0.1 rmedeiro.42.fr

3) Check the Configuration: ping rmedeiro.42.fr

   Expected output: PING rmedeiro.42.fr (127.0.0.1)

The /etc/hosts file acts as a local DNS table.
When the browser requests https://rmedeiro.42.fr the operating system automatically translates: rmedeiro.42.fr → 127.0.0.1
which points to the local machine.
This allows NGINX to receive requests for the domain required by the subject without needing a real public DNS record.

If you access the website from your host machine (outside the VM), you must also add 127.0.0.1 rmedeiro.42.fr to the host 
machine's hosts file.
Otherwise, the browser running on the host will not be able to resolve the domain name.

Create Volumes on the Host Machine

In this project, the VM is considered the host machine for Docker.
When the subject says /home/login/data on the host machine, it means /home/rmedeiro/data inside the Debian VM, because
Docker is running inside the VM.
So, the subject explicitly requires both persistent storages in /home/rmedeiro/data.

Create the required folders:  mkdir -p /home/rmedeiro/data/mariadb
                              mkdir -p /home/rmedeiro/data/wordpress

Check if they were created correctly: ls -la /home/rmedeiro/data

Containers are temporary by nature. If a container is removed, its internal filesystem is also removed.
However, MariaDB and WordPress need persistent data:

MariaDB   → database files
WordPress → website files

So we store them in Docker named volumes mapped to: /home/rmedeiro/data/mariadb
                                                    /home/rmedeiro/data/wordpress

This way, if containers are deleted and recreated, the data is not lost.
That's why we need to create them.

Create the .env file

The .env file stores environment variables used by Docker Compose and by your containers.
Instead of hardcoding values directly in scripts or in docker-compose.yml, we define them once in .env.

We create the .env file inside the srcs folder: vim srcs/.env

What must be inside .env? For the mandatory part, we need: DOMAIN_NAME
										                   MYSQL_DATABASE
										  				   MYSQL_USER
										                   WP_TITLE
										                   WP_ADMIN_USER
										                   WP_ADMIN_EMAIL
										                   WP_USER
										                   WP_USER_EMAIL

What should NOT be inside .env? We do not put passwords in .env. We avoid this: MYSQL_PASSWORD=1234
                                                                                WP_ADMIN_PASSWORD=1234

Passwords should be stored in Docker secrets instead. Why? The subject says credentials, passwords and API keys 
should not be stored directly in the Git repository. The .env file is useful for configuration, but it is not 
ideal for secrets.

Create Docker Secrets

Create the secrets files: echo "root42" > secrets/db_root_password.txt
						  echo "db42" > secrets/db_password.txt
						  echo "wp42" > secrets/wp_admin_password.txt
						  echo "user42" > secrets/wp_user_password.txt

What each secret is for?

db_root_password.txt - Password for the MariaDB root user.

db_password.txt - Password for the normal MariaDB user used by WordPress.

wp_admin_password.txt - Password for the WordPress administrator account.

wp_user_password.txt - Password for the normal WordPress user.

Set Secure Permissions on Secrets:  chmod 600 secrets/*

What does chmod 600 mean? 6 - owner can read and write
						  0 - group has no permissions
                          0 - others have no permissions

So only the owner can read and edit the files.
In practice: -rw------- db_password.txt

So only the owner can read and edit the files.

Why do we use it? Because secrets contain passwords. This prevents other users on the system from reading them.
Even though this is a local VM, it is good practice and matches the security expectation of the subject.

Create .gitignore

We create the .gitignore file at the root of the project: vim .gitignore

We add: secrets/
        srcs/.env

Why ignore secrets/? Because it contains passwords. These files must never be pushed to GitHub.

Why ignore srcs/.env? Because .env may contain machine-specific configuration. Even if we avoid passwords 
there, it can still contain local values such as: DOMAIN_NAME
												  database names
												  usernames
												  emails

For the evaluator, we can provide an example file separately if needed, such as: srcs/.env.example
but the real .env should stay local.
